# Camaraderie

Camaraderie is a Meetup-inspired event platform for discovering, creating, and joining local events. It combines secure account management, event-scoped roles, real-time participant chat, an API for external integrations, offline support, and production monitoring in one containerized web application.

## Description

### Goal

Camaraderie helps people form communities around shared interests and activities. Registered users can browse upcoming events, create and manage their own events, join events organized by others, and communicate with fellow participants in an event-specific live chat.

The application was designed as a multi-user system: authentication, event membership, authorization, chat rooms, persistence, and monitoring are handled by separate layers and services.

### Key features

- Secure registration and login with Argon2id password hashing and signed JWT session cookies.
- Event discovery with pagination and participant information.
- Event creation, editing, deletion, capacity management, and PNG cover images.
- Event-scoped `admin` and `member` roles with protected management actions.
- Join, leave, list-participant, and remove-participant workflows.
- Persistent, participant-only real-time chat using WebSockets.
- Versioned public event API secured by API keys and rate limiting.
- Interactive OpenAPI documentation generated by Huma and rendered with Scalar.
- Installable Progressive Web App with offline-aware navigation and caching.
- Prometheus metrics, alerting rules, and provisioned Grafana dashboards.
- Privacy Policy and Terms of Service pages linked from the application.
- Support for current stable Chrome, Firefox, and Brave releases.

### Authentication model

Passwords are never stored directly. The backend validates registration input and hashes passwords with Argon2id. After a successful login, it creates a signed JWT containing the user identifier and expiration information. The token is stored in a cookie named `jwt` with `HttpOnly`, `Secure`, `SameSite=Strict`, and `/api` path restrictions. Protected backend middleware verifies the signature and refreshes valid sessions.

## Architecture

This project is a containerized full-stack web application. It follows a service-oriented architecture where the frontend, backend, database, and reverse proxy are isolated into separate components and communicate over internal networks.

When a user visits our website, the DNS resolves the domain name to the IP address of the VPS where the ft_transcendence project is hosted. The browser then sends HTTP(S) requests to that IP, which is handled by the server running on the VPS.
In our setup, these requests first reach the system-level reverse proxy, Caddy, which listens on ports 80 (HTTP) and 443 (HTTPS). The purpose of this reverse proxy layer is to manage incoming traffic for multiple services running on the same machine (example: there might be another different website running on the same machine) and route requests to the appropriate application based on the domain name and path.

For the domain ft-transcendence.zaper.io, the system Caddy acts as an entry-level reverse proxy and forwards traffic to the Transcendence-specific Caddy instance. The Transcendence Caddy then routes requests internally, for example directing /api/* requests to the backend API while serving frontend traffic for all other routes.

When a user presses a button that triggers an API call, the request is sent from the browser and routed through Caddy to the backend service, typically via a path such as /api/*.

An interesting aspect of the architecture is the potential communication between the frontend and backend containers which the diagram above shows. In most cases, these services do not communicate directly. Instead, all client-triggered requests pass through the reverse proxy.

However, there is one important exception: when using Server Side Rendering (SSR), the frontend container itself may act as a server. In this case, during Server-Side Rendering (SSR), the Next.js server can make internal HTTP requests to the backend API before sending the fully rendered HTML to the client.

### Architecture during development

During development, the architecture differs slightly from production. Running the entire application inside containers can be a little tedious and more complex. Instead, we use a hybrid approach: **local development with dependencies running in containers**.

While developing entirely inside containers offers better reproducibility and helps avoid “it works on my machine” issues, it requires more setup and can introduce additional complexity. For most development tasks, working directly on your local machine is faster, more convenient, and feels more natural (still comes with its own pitfalls though).

> **Note:** I only know one transcendence repository which does development inside of containers. If you are interested in what that looks like, you can take a look at it here https://github.com/cubernetes/ft-transcendence. The key aspect is the use of Dockers file-watching functionality (in compose it's the use of the watch directive).

## Database schema

We generally use soft deletes. Deleted entries remain in the database, and their `deleted_at` column is updated to indicate that the application should ignore them.

### users

This is the database representation of users.

| Column | Data Type | Nullable |
|---|---|---|
| id | bigint | NO |
| created_at | timestamp with time zone | YES |
| updated_at | timestamp with time zone | YES |
| deleted_at | timestamp with time zone | YES |
| name | text | YES |
| email | text | YES |
| password_hash | text | YES |

### events

This is the database representation of events.

| Column | Data Type | Nullable |
|---|---|---|
| id | bigint | NO |
| created_at | timestamp with time zone | YES |
| updated_at | timestamp with time zone | YES |
| deleted_at | timestamp with time zone | YES |
| title | character varying | NO |
| description | text | YES |
| start_time | timestamp with time zone | NO |
| duration | smallint | NO |
| location_name | character varying | YES |
| location_address | character varying | YES |
| max_capacity | bigint | NO |
| image_path | character varying | YES |

### event_users

This join table describes relationships between events and users. An entry means that the referenced user has joined the referenced event. The `role` column can be `admin` or `member`.

| Column | Data Type | Nullable |
|---|---|---|
| id | bigint | NO |
| created_at | timestamp with time zone | YES |
| updated_at | timestamp with time zone | YES |
| deleted_at | timestamp with time zone | YES |
| user_id | bigint | NO |
| event_id | bigint | NO |
| role | text | NO |

### messages

This is the database representation of persistent chat messages. Each message is associated with an event through `event_id` and its sender through `user_id`. The sender's name is retrieved from `users` when the API constructs a `MessageDTO`; it is not duplicated in this table.

| Column | Data Type | Nullable |
|---|---|---|
| id | bigint | NO |
| created_at | timestamp with time zone | YES |
| updated_at | timestamp with time zone | YES |
| deleted_at | timestamp with time zone | YES |
| event_id | bigint | NO |
| user_id | bigint | NO |
| content | text | NO |

### api_keys

This is the database representation of API keys used for the public API. Only hashes and revocation state are stored; raw API keys are not persisted.

| Column | Data Type | Nullable |
|---|---|---|
| id | bigint | NO |
| created_at | timestamp with time zone | YES |
| updated_at | timestamp with time zone | YES |
| deleted_at | timestamp with time zone | YES |
| key_hash | text | YES |
| revoked | boolean | YES |

## Instructions

### Prerequisites

Required:

- Git
- Docker Engine or Docker Desktop
- Docker Compose v2, available through `docker compose`

Optional for development outside containers:

- Go 1.26 or a compatible version from [`backend/go.mod`](./backend/go.mod)
- Node.js 24 or a version compatible with [`frontend/package.json`](./frontend/package.json)
- `just` for repository helper recipes

### Configure the environment

Clone the repository and enter it:

```bash
git clone https://github.com/zaperfish/ft_transcendence.git transcendence
cd transcendence
```

Create the local environment file:

```bash
cp .env.example .env
```

Fill every blank value in `.env`. Important settings include:

| Variable | Purpose |
|---|---|
| `DOMAINS` | Caddy site addresses; for local use, `https://localhost http://localhost`. |
| `NODE_ENV` | Use `production` for the containerized evaluation build. |
| `JWT_KEY` | Secret used to sign JWTs. Generate a new value, for example with `openssl rand -base64 32`. |
| `ADMIN_PASSWORD` | Protects API-key administration endpoints. |
| `GRAFANA_ADMIN_USER` / `GRAFANA_ADMIN_PASSWORD` | Grafana administrator credentials. |
| `POSTGRES_USER` / `POSTGRES_PASSWORD` / `POSTGRES_DB` | Database credentials and database name. |
| `POSTGRES_HOST` | Use `postgres` for Docker Compose. |

Do not commit `.env`. The repository tracks only `.env.example`, which must contain placeholders rather than deployment credentials.

### Start the complete application

Build and start every service with one command:

```bash
docker compose up --build -d
```

Check their state:

```bash
docker compose ps
```

The expected local entry points are:

| Service | URL |
|---|---|
| Camaraderie | `https://localhost:7443` |
| OpenAPI document | `https://localhost:7443/openapi.json` |
| Interactive API documentation | `https://localhost:7443/docs` |
| Grafana | `https://localhost:7443/grafana/` |
| Prometheus | `https://localhost:7443/prometheus/` |

### Trust the local HTTPS certificate

Caddy creates a local certificate authority. Export its root certificate:

```bash
docker compose cp caddy:/data/caddy/pki/authorities/local/root.crt ./caddy-local-root.crt
```

Import `caddy-local-root.crt` into the browser's trusted certificate store and restart the browser. Only trust a certificate exported from your own local Caddy container.

### Stop the application

```bash
docker compose down
```

Add `--volumes` only when intentionally deleting the local database, uploads, and monitoring data.

## Usage

1. Open `https://localhost:7443` in a supported browser.
2. Register with a unique username and email and a password satisfying the displayed policy.
3. Log in and browse upcoming events.
4. Create an event and optionally select a PNG cover image.
5. Open an event to edit it as its administrator or join it as a member.
6. Enter the event chat after joining and exchange live messages with another connected participant.
7. Use Settings to update the account email, change the password, or delete the account.
8. Open Grafana to demonstrate provisioned dashboards and Prometheus alerts.

### Browser validation

Core workflows are intended to behave consistently on the latest stable versions of:

- Google Chrome — mandatory reference browser
- Mozilla Firefox — additional browser
- Brave — additional browser

The browser check covers registration/login, navigation, event CRUD, membership, image display, WebSocket chat, responsive layouts, legal pages, and PWA behavior where supported by the browser. Any browser-specific limitation discovered during final testing should be recorded here before evaluation.

### 1. Frameworks for frontend and backend — Major, 2 points

**Why:** The application needs consistent routing, validation, UI composition, middleware, and API contracts across a multi-person team.

**Implementation:** Next.js, React, TypeScript, and Tailwind provide the frontend architecture. Chi provides backend routing and middleware, while Huma provides typed request handling, validation, OpenAPI generation, and Scalar documentation.

**Ownership:** `lmiehler`, `yingzhan`, `alneuman`, `dlemaire`, and `shutan`.

### 2. Real-time WebSocket features — Major, 2 points

**Why:** Participants need immediate communication within their event community.

**Implementation:** Each event has an isolated backend room. Connections are authenticated and restricted to participants. Messages are validated, persisted, and broadcast efficiently; slow or disconnected clients are removed without blocking the room. The frontend merges stored history with live messages and displays connection status.

**Ownership:** `dlemaire`.

### 3. Public API — Major, 2 points

**Why:** External clients should be able to integrate with event data without browser session cookies.

**Implementation:** `/api/v1/events` exposes versioned GET, POST, PUT, PATCH, and DELETE operations. API keys are generated through administrator-protected endpoints, stored as hashes, and checked by middleware. Global rate limiting protects the service. Huma publishes the OpenAPI contract and Scalar renders interactive documentation.

**Ownership:** `lmiehler`.

### 4. Prometheus and Grafana monitoring — Major, 2 points

**Why:** A deployed multi-service application needs visibility into availability and runtime behavior.

**Implementation:** Prometheus scrapes the Go metrics endpoint every five seconds and evaluates rules for downtime, restarts, and high goroutine counts. Grafana uses a provisioned Prometheus data source and custom dashboards covering garbage collection, memory, goroutines, threads, `GOMAXPROCS`, and CPU usage. Grafana is served over HTTPS and requires authentication.

**Ownership:** `dlemaire`. Demonstration details are in [`monitoring/runtime-pressure.md`](./monitoring/runtime-pressure.md).

### 5. Advanced permissions — Major, 2 points

**Why:** Organizers and attendees require different capabilities and backend-enforced access control.

**Implementation:** Memberships contain `admin` or `member` roles. Event creators become admins automatically. Only admins can update or delete their event, manage its image, or remove participants. Members can join, leave, view permitted data, and use participant-only chat. The frontend shows actions according to the current membership role while the backend remains the authority.

**Ownership:** `alneuman`, `lmiehler`, and `yingzhan`.

### 6. Organization system — Major, 2 points

**Why:** Camaraderie needs managed communities in which users can gather around a shared purpose and perform role-specific actions.

**Domain mapping:** In Camaraderie, an event community is the organization unit. The `events` table stores the managed organization resource and `event_users` stores its membership and roles.

**Implementation:** Users can create, view, update, and delete event communities; join them; leave them; list their participants; and, as administrators, remove participants. Members can read event information and participate in its chat, while administrators receive management actions. This provides organization CRUD, membership management, organization views, and multiple organization-scoped actions.

**Ownership:** `alneuman`, `lmiehler`, and `yingzhan`.

### 7. ORM — Minor, 1 point

**Why:** Typed models and repository abstractions reduce repeated SQL and keep persistence logic consistent.

**Implementation:** GORM maps the domain models to PostgreSQL, performs startup migrations, builds queries and transactions, and supports soft deletion.

**Ownership:** `alneuman` and `lmiehler`.

### 8. Progressive Web App — Minor, 1 point

**Why:** Users should be able to install Camaraderie and retain useful navigation and cached information when connectivity is interrupted.

**Implementation:** The application includes a web manifest, application icons, a production service worker, runtime caching strategies, offline responses, an offline banner, and standalone display configuration.

**Ownership:** `yingzhan`.

### 9. Additional browsers — Minor, 1 point

**Why:** Users should receive a consistent experience outside the mandatory Chrome environment.

**Implementation:** Responsive layouts, forms, navigation, event management, WebSocket chat, and legal pages are supported on current stable Chrome, Firefox, and Brave releases. Firefox and Brave are the two additional module browsers.

**Ownership:** `yingzhan`, `dlemaire`, and `shutan`.
